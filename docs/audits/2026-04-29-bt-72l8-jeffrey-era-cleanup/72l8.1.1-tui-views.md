# 72l8.1.1: TUI views ghost-feature audit

Sub-area of bt-72l8.1. Per-view classification of every top-level interactive
TUI surface (ViewMode enum + the modal overlays that present a "view-like"
panel, e.g. flow matrix, label dashboard).

## Methodology

Hybrid:

1. **Read-only source inspection**. Enumerated all top-level views from the
   `ViewMode` enum in `pkg/ui/model.go:83-95` and modal overlays from the
   `ModalType` enum at lines 101-118. Cross-referenced view-rendering dispatch
   in `pkg/ui/model_view.go:48-130` and view-entry keybindings in
   `pkg/ui/model_update_input.go`.
2. **Robot-mirror live tests** in the bt workspace itself (cwd
   `C:\Users\sms\System\tools\bt`):
   - `bt robot insights` -> WORKS (large JSON, real metrics)
   - `bt robot graph` -> WORKS
   - `bt robot plan` -> WORKS
   - `bt robot history` -> ERRORS: "no beads file found in
     C:\Users\sms\System\tools\bt/.beads/" (current Dolt-only repo has no
     `.beads/beads.jsonl`)
   - `bt robot labels health|flow` -> WORKS
3. **Source-side data-path verification**. For each view, traced what the
   render path opens / reads / depends on, then checked whether that source
   exists in the current workspace. Workspace is Dolt-only (`.beads/dolt/`
   present, no `.beads/*.jsonl` except internal `correlation_feedback.jsonl`
   and `interactions.jsonl`).

Important caveat: launching `bt` interactively was avoided (blocks the
session). Behavior of views entered via key bindings was inferred from the
update path that mutates `m.mode` plus the data path the renderer reads.

## GHOST views (high priority)

### ViewSprint (most severe)

- **Why ghost**: Two independent failure modes stack:
  1. Data path opens `.beads/sprints.jsonl` (`pkg/loader/sprint.go:28`); upstream
     beads has no sprints concept; the file has never existed in this repo and
     never will under Dolt-only beads.
  2. No keypress sets `m.mode = ViewSprint` in production code paths -
     `pkg/ui/model_update_input.go` has no entry. Only
     `pkg/ui/sprint_view_keys_test.go` and `pkg/ui/context_test.go` set it.
- **Recommendation**: **retire**. Cross-reference the existing decision bead
  bt-z5jj (P3, "rebuild against Dolt or retire"). MEMORY.md records this.
  Until that decision lands, the keybind dispatcher and renderer should be
  feature-flagged off so nobody trips into an unreachable+broken view.

### ViewHistory (effectively ghost on Dolt-only repos)

- **Why ghost**: TUI path goes through `correlation.NewCorrelator` which parses
  `git log` diffs of `.beads/*.jsonl`. In a Dolt-only repo (current workspace
  being the canonical example - bt itself), no such files exist after commit
  `90d8432d`, so the report is empty or stale. `bt robot history` already
  errors out on this workspace ("no beads file found"). The TUI doesn't error
  but renders an empty / pre-deletion-only view.
- **Recommendation**: **rebuild against Dolt** (`dolt_log` +
  `dolt_history_issues`) - already tracked as **bt-08sh** (P2). This audit
  should bump that bead's priority to P1 since the TUI surface is more
  user-facing than the robot subcommand.
