# bt-u8iz Implementation Plan: per-bead History via path-lookup primitive

**Status**: draft, 2026-05-06
**Bead**: bt-u8iz (P2, area:correlation, area:tui)
**Follow-ups blocked by this**: bt-m10p (paired beads), bt-7l34 (firehose)
**Coordinates with**: bt-08sh (correlator Dolt migration)

## Overview

History view in global mode currently fails to populate because `correlation.NewCorrelator(cwd, ...)` is called with the user's cwd, not the project's git root. When bt is launched outside a project (e.g. from `~`), there is no git context. bt-ezk8 shipped a graceful empty-state message; bt-u8iz ships the actual data path.

**The reframe** (locked 2026-05-06): the natural unit of history is the *bead*, not the *view-filter*. Every bead carries its project prefix; the prefix uniquely identifies which git repo to query. Aggregation problem dissolves — each `H`-press is a single-project query, just driven by the bead's identity instead of the active filter.

**The primitive**: `~/.bt/projects.json`, a user-global registry mapping `prefix → {path, last_seen}`. Auto-populated when bt launches inside a project's git tree. Lazy-stat'd on lookup, falls through to bt-ezk8-style empty-state on stale entries.

**Surface area**: one new package (`pkg/projects/`), a 5-line extension to `workspace.LoadResult` (carry `AbsPath`), three-mode auto-stamp at `cmd/bt/root.go::loadIssues` (cwd / workspace / global), and a refactor of `enterHistoryView` to resolve path via registry. Three commits.

## Architecture (L8 lens)

### Primitive vs feature

The registry is a **primitive**, not a History feature. Naming it that way pre-commits to:

- A clean public interface (`Lookup`, `Stamp`, `Entries`) that other features can consume without depth-of-implementation knowledge.
- Zero coupling to the correlation package, the UI, or any specific view.
- A package boundary at `pkg/projects/` — peer to `pkg/correlation/`, `pkg/agents/`, `pkg/baseline/`. Not buried inside `pkg/ui/` or `internal/`.

This matters because the same primitive will service bt-m10p (paired beads need both prefixes' paths), bt-7l34 (firehose iterates all entries), and likely future affordances (jump-to-files, open-in-editor, cross-project grep). Build it once, build it right.

### Coupling decisions

**Path resolution happens at the call site, not inside the correlator.** Today `correlation.NewCorrelator(cwd, beadsPath)` is called from `pkg/ui/model_modes.go:98` (TUI History entry) and `cmd/bt/robot_history.go` (4 robot variants) and `cmd/bt/burndown.go:458`. Refactoring the correlator to take a `prefix` argument and look up paths internally would conflate two concerns: the correlator's job is "given a path, build a history report"; finding the path is upstream.

The right shape: `enterHistoryView` resolves prefix → path via the registry, then passes the resolved path to `NewCorrelator(path, beadsPath)`. Correlator's signature is unchanged. This keeps the registry composable — robot subcmds and burndown can adopt registry resolution later without correlator-side coupling.

### What does "a project's path" mean?

A subtle architectural question: in the typical case `.beads/` lives at git toplevel, so "the project's path" is unambiguous. In a monorepo where `.beads/` is in a subdirectory (e.g. `repo/services/billing/.beads/`), git toplevel and the project's logical home diverge. The consumer's intent ("show me what changed in this project") cleanly maps to one or the other, not both:

- **Store git toplevel** (v1 choice): consumer runs `git log` and gets every commit in the repo, including those touching unrelated subdirs. Slight noise; matches today's `correlation.NewCorrelator(repoPath, ...)` call shape with zero correlator change.
- **Store `.beads/`-parent + pathfilter**: cleaner per-project log, but requires `git log -- <subdir>` invocation shape — a different correlator call.

V1 stores git toplevel. Deliberate simplification, documented here as reversible. Trigger for the future bead: real monorepo dogfood surfaces noisy History output; at that point, add `Entry.Subdir` to the registry shape and pathfilter at the consumer.

### Future-proofing for bt-08sh

bt-08sh migrates the correlator from JSONL+git-diff witness to `dolt_log` + `dolt_history_issues`. After bt-08sh, the correlator no longer needs filesystem paths to *build* a history report — Dolt has the data.

But the registry stays useful:

- **File-level git diffs**: `dolt_log` records bead edits; it doesn't record source-code changes co-committed with bead edits. To show "files changed" alongside bead history, bt still needs the project's git path for `git show <sha>`.
- **Jump-to-files / open-in-editor**: future affordances need filesystem paths regardless.
- **bt-7l34 firehose, filesystem-walk path**: if bt-7l34 ships before bt-08sh, registry is the data source.

Conclusion: the registry doesn't get ripped out by bt-08sh — it becomes optional infrastructure for the bead-history rendering path, and load-bearing for adjacent features. Build it now, no rip-out cost.

### Testability of the primitive in isolation

The package boundary lets `pkg/projects/` be unit-tested without touching git, correlation, or the UI:

- Round-trip serialization (write registry, read it back).
- Stamping updates `last_seen` and overwrites `path` on relaunch from a different location.
- Lookup with present, missing, and stale entries.
- Concurrent writes from multiple bt sessions: atomic temp+rename prevents partial-file corruption; lost-update race accepted as v1 (self-heals on next launch). Test verifies file is never written in a partial state.

Integration tests in `pkg/ui/history_test.go` then verify the end-to-end behavior with a temp registry.

| 3 | History consumer refactor | `pkg/ui/history.go`, `pkg/ui/model_modes.go`, `pkg/ui/history_test.go` | `feat(tui): per-bead History resolution via projects registry (bt-u8iz)` |

Each phase is independently testable, builds and passes vet+test in isolation. Phase 1 ships a usable primitive even before Phases 2-3 land. Phase 3 fully closes the bead.

### Sequence rationale

- **Phase 1 first**: smallest, no dependencies, clean test surface. Can be cherry-picked or reverted independently.
- **Phase 2 before Phase 3**: Phase 3's tests need *some* registry data to exist; Phase 2 is what populates it organically. Strict ordering not load-bearing — Phase 3 tests can also seed registry directly via `pkg/projects/` API.
- **Phase 3 closes the bead**: only after the user-facing behavior works end-to-end.

### Risks

| Risk | Mitigation | Probability |
|---|---|---|
| Concurrent writes to `~/.bt/projects.json` from parallel bt sessions lose updates (last-write-wins) | Atomic temp+rename prevents file corruption (partial writes), but does NOT prevent lost updates: A loads {X}, B loads {X}, A stamps Y, B stamps Z, A saves {X,Y}, B saves {X,Z} — Y is lost. Accept the race in v1 + document: lost stamps self-heal on next bt launch in the affected project (re-stamp re-fires). The race window is small (ms-scale read-modify-write); concurrent stamps to the same prefix from different paths are vanishingly rare in practice | Low |
| `git rev-parse --show-toplevel` shells out and is slow on Windows for first launch | Cache result in-process (not just registry); shell out at most once per launch | Low |
| `HistoryModel` refactor breaks existing tests that use `SetContext(HistoryContext{...})` | Phase 3 extends `HistoryContext` rather than replacing it; existing tests continue to work | Low |
| `--workspace` multi-repo stamping has different failure modes than cwd-mode (one repo's git toplevel resolves while another's doesn't) | Stamp logic skips repos where `gitToplevel` resolution fails; logs via pkg/debug; never blocks launch. `TestAutoStamp_WorkspaceFailedRepo` covers the partial-success case | Low |
| Registry contains stale path for currently-active project; auto-stamp on launch should refresh it | Phase 2 always overwrites `path` and updates `last_seen` on every launch | Low |
| Two clones of the same project (same prefix, different filesystem paths) → last-write-wins | Acceptable v1 behavior; document in registry comment + plan | Low |
| Pair-detection collisions cause the wrong project's history to render | Out of scope for u8iz — no pair detection in v1; bead's prefix is taken at face value | N/A |

### Completion criteria (gating to close bt-u8iz)

1. `~/.bt/projects.json` populates automatically when bt is launched inside a project's git tree.
2. In global mode (`bt` from `~`), pressing `H` on a bead from project X produces project X's actual git history. Verified manually + e2e.
3. With no bead under cursor, History falls back to the active filter (single-project case) and uses that project's path. Stale-entry case shows the bt-ezk8 empty-state message.
4. `go build ./... && go vet ./... && go test ./...` all green.
5. Audit doc (this plan) updated with "shipped" stamp; CHANGELOG.md entry added.

## User experience (PdM lens)

### Resolution priority (user-visible)

When the user views History, bt resolves the project path in this order:

1. **Cursor's current bead** → use bead's prefix → registry → path → git log. Most natural; the cursor's bead is "what the user is looking at."
2. **Active single-project filter** (no cursor or cursor not on a bead row) → use filter's prefix.
3. **No usable context** (no cursor, no filter, multi-project filter, prefix not in registry, registry path stale) → bt-ezk8-style empty-state message, refined to mention cursor selection.

### Edge cases worth naming explicitly

| Scenario | Behavior |
|---|---|
| Global mode, cursor on bt bead, bt is in registry | History populates with bt's git log |
| Global mode, cursor on bt bead, bt is NOT in registry (never launched inside) | Empty-state message: "bt's filesystem path is not known yet — launch bt from inside the project once to register it" |
| Global mode, no cursor, single-project filter on bt | History populates with bt's git log |
| Global mode, no cursor, multi-project filter | Empty-state message: "select a single bead or apply a single-project filter" |
| Global mode, cursor moves from bt bead to bd bead | History refreshes against bd's repo (UX call: probably debounce so rapid scrolling doesn't thrash; v1 can update on dwell) |
| Local mode (bt launched inside project) | Behavior unchanged from today; cwd is authoritative; registry stamping happens as side effect |
| `~/.bt/projects.json` doesn't exist | `Lookup` returns "not found"; first auto-stamp creates the file |
| `~/.bt/projects.json` is corrupt JSON | Treat as empty registry; log a debug message; subsequent stamp rewrites the file. Don't panic |
| Project moved (path stale, not present) | Lookup falls through; empty-state with hint: "registered path no longer exists — launch bt from the new location to refresh" |
| Project deleted | Same as moved: stale entry, fall through, hint to refresh |
| Project's `.beads/` is in a subdirectory of git toplevel (monorepo case) | v1 stamps git toplevel; History runs `git log` against toplevel, including commits from unrelated subdirs (slight noise). v2 trigger: if real monorepo dogfood surfaces, follow-up bead adds `Entry.Subdir` and pathfilters `git log -- <subdir>`. See "What does 'a project's path' mean?" in Architecture |

### Discoverability

- Auto-stamp is silent (no notification on launch). Users discover via "it just works" once they've launched bt in each project.
- Stale-entry messages should be specific: don't say "history unavailable" — say "registered path for `bt` no longer exists; launch bt from the new location to refresh." Verbatim path inclusion makes the recovery action obvious.
- Add a one-line entry to AGENTS.md "Key Design Constraints" referencing the registry, so future agents know where it lives.

### Cursor-driven refresh: UX call to make during Phase 3

Two reasonable behaviors:

- **Update on dwell**: scrolling fast through bead list doesn't refresh; resting cursor on a bead for ~200ms triggers refresh. Smoother visually; small perceived lag.
- **Update on selection**: only refreshes when `H` is pressed (current behavior); cursor movement doesn't trigger.

Recommend **update on selection** for v1. Consistent with current behavior, smaller blast radius. Dwell-refresh is a separate UX bead if dogfood proves it's wanted.

### Dogfood plan

After Phase 3 ships:

1. Launch bt in `bt`, in `bd`, in any other beads-init project user has. Verify each project gets a registry entry.
2. From `~`, launch `bt --global`. Cursor on bead from each project, press `H`. Verify history populates.
3. Move one project to a new path. Launch bt from new path. Verify registry path updates.
4. Move a project, but don't launch bt from new path. Try to view its history from `~`. Verify empty-state message names the stale path.
5. Single-project filter case (no cursor): apply filter to bt, press `H` from blank cursor area. Verify history populates.
6. Multi-project filter case: apply filter to bt+bd, no cursor. Verify empty-state message says "select a single bead or single-project filter."

Each of these should produce a one-line capture: "✓ behavior matches expectation" or "✗ <symptom> — file followup."

### Phase 1: Registry primitive

- `pkg/projects/registry_test.go` — unit tests

Public API surface (initial):

```go
package projects

type Entry struct {
    Path     string    `json:"path"`
    LastSeen time.Time `json:"last_seen"`
}

type Registry map[string]Entry

// Load reads the registry from disk. Returns an empty Registry (not error)
// if the file doesn't exist. Returns an empty Registry + debug-logged error
// if the file is corrupt JSON.
func Load(path string) (Registry, error)

// Save writes the registry to disk via atomic temp+rename. Creates parent
// directory if needed.
func Save(path string, r Registry) error

// Stamp records (or updates) a (prefix, abs-path) entry with now() as
// last_seen. Returns the updated Registry.
func Stamp(r Registry, prefix, absPath string, now time.Time) Registry

// Lookup returns the Entry for prefix and whether it's currently valid
// (path exists and contains .git/). Stale entries return (entry, false).
// Missing entries return (Entry{}, false).
func Lookup(r Registry, prefix string) (Entry, bool)

// DefaultRegistryPath returns ~/.bt/projects.json, mirroring events.DefaultPersistPath().
func DefaultRegistryPath() (string, error)
```

Convenience operations on the package level for the call site:

```go
// LookupAndValidate is the registry consumer's one-liner:
// load, look up, return path or empty.
func LookupAndValidate(prefix string) (path string, ok bool)
```

### Phase 2: Auto-stamp wiring

**Entry point (recon resolved)**: bt has a single chokepoint where project detection completes. `cmd/bt/root.go::rootPersistentPreRun` runs for every command (cobra_robot.go calls it from its own PersistentPreRunE, so robot subcmds flow through too). `cmd/bt/root.go::loadIssues` runs once per launch when data is needed and populates `appCtx.currentProjectDB` (cwd mode) or `appCtx.workspaceInfo` (workspace mode).

**Stamp site**: end of `loadIssues()`. Three branches by launch mode:

- **cwd mode** (no `--workspace`, no `--global`): one stamp. `(appCtx.currentProjectDB, gitToplevel(cwd))`.
- **`--workspace` mode**: N stamps, one per loaded repo. `--workspace` is multi-repo by design (per `pkg/workspace/loader.go::LoadResult` carrying per-prefix issue lists, `LoadSummary.RepoPrefixes []string`). Walk the `[]LoadResult` returned from `workspace.LoadAllFromConfig`, stamp each `(Prefix, gitToplevel(AbsPath))` — same git-toplevel resolution as cwd mode for consistency with the correlator's `repoPath` semantics. Repos where `gitToplevel` fails are skipped.
- **`--global` mode**: zero stamps from launch. Global mode by definition isn't "inside" any single project; the registry is consumed but not populated by global launches.

**Prerequisite**: `workspace.LoadResult` (loader.go:17) currently carries `RepoName, Prefix, Issues, Error` but not absolute path. Add `AbsPath string` as a pre-Phase-2 change (~5 lines in `pkg/workspace/`): pair prefix↔path at the LoadResult level so the stamp site doesn't need to re-walk the workspace config. This is a small, principled extension — the loader already resolves `workspaceRoot + repo.Path` internally per cobra_export.go:322-326; it's just not surfaced on the result.

This keeps the single-chokepoint shape: project detection happens at command startup; the registry stamps once at the chokepoint where detection has succeeded. Each launch mode's stamping logic lives in one place (end of `loadIssues`), but branches by mode because each mode has different detection semantics. Other beads-dir resolvers (`cmd/bt/burndown.go`, `cmd/bt/pages.go`, `cmd/bt/cli_misc.go`) are downstream consumers — they flow through `loadIssues` and don't need their own stamp call.

Files touched:
- `pkg/workspace/loader.go` — extend `LoadResult` with `AbsPath string` (5-line change, no new test failures expected; existing tests construct `LoadResult` literals so they pick up zero-value AbsPath cleanly)
- `pkg/workspace/loader_test.go` — one new assertion: `LoadResult.AbsPath` populated after `LoadAll`
- `cmd/bt/root.go` — add the stamp logic at the end of `loadIssues()`, branched by mode (cwd / workspace / global per the entry-point section above)

`--workspace <path>` flows through `loadIssues` via the existing `flagWorkspace` branch (root.go:182-216). Stamp site is downstream of both `flagWorkspace` and the cwd-default branches; mode-specific stamping selects the right behavior.

Resolving git root for stamping:

```go
func gitToplevel(startPath string) (string, error) {
    // Reuse correlation.IsInsideWorkTree pattern, OR shell out:
    cmd := exec.Command("git", "rev-parse", "--show-toplevel")
    cmd.Dir = startPath
    out, err := cmd.Output()
    if err != nil { return "", err }
    return strings.TrimSpace(string(out)), nil
}
```

Or use `correlation.IsInsideWorkTree` + walk upward to find `.git/`. Either works; first option matches existing burndown.go patterns.

Stamp happens once per launch (load → stamp each prefix → save once), async-fire-and-forget if it errors (don't block launch on stamp failure). All errors log via `pkg/debug` and return silently. The cwd path stamps a single `(prefix, gitToplevel(cwd))` pair; workspace path iterates `[]LoadResult` and stamps each `(Prefix, gitToplevel(AbsPath))` pair where resolution succeeds; global path is a no-op.

Path resolution in `enterHistoryView` (rewritten):

```go
func (m *Model) enterHistoryView() {
    cwd, err := os.Getwd()
    if err != nil {
        m.setStatusError("Cannot get working directory for history")
        return
    }

    ctx := m.historyContext()  // now includes CursorPrefix
    repoPath := resolveHistoryPath(ctx, cwd)  // NEW helper

    beads := m.buildBeadInfos()  // unchanged
    correlator := correlation.NewCorrelator(repoPath, m.data.beadsPath)
    report, err := correlator.GenerateReport(beads, correlation.CorrelatorOptions{Limit: 500})
    if err != nil {
        m.setStatusError(fmt.Sprintf("History load failed: %v", err))
        return
    }

    m.historyView = NewHistoryModel(report, m.theme)
    m.historyView.SetContext(ctx)
    m.historyView.SetSize(m.width, m.height-1)
    m.mode = ViewHistory
    m.focused = focusHistory
    m.setStatus(fmt.Sprintf("Loaded history: %d beads with commits", report.Stats.BeadsWithCommits))
}

// resolveHistoryPath implements the cursor → filter → cwd priority order.
// Returns cwd as the safe fallback that preserves bt-ezk8 behavior when
// registry has nothing useful to say.
func resolveHistoryPath(ctx HistoryContext, cwd string) string {
    if ctx.CursorPrefix != "" {
        if path, ok := projects.LookupAndValidate(ctx.CursorPrefix); ok {
            return path
        }
    }
    if len(ctx.ActiveProjects) == 1 {
        if path, ok := projects.LookupAndValidate(ctx.ActiveProjects[0]); ok {
            return path
        }
    }
    return cwd  // bt-ezk8 graceful degradation: empty-state message handles this
}
```

The cursor's prefix derivation (recon resolved): bt's existing pattern at `pkg/ui/model_update_data.go:68-72` uses `m.list.SelectedItem()` returning nil when no selection exists; otherwise type-asserts to `IssueItem` to access `.Issue.ID`. Mirror this in `historyContext()`:

```go
ctx := HistoryContext{WorkspaceMode: m.workspaceMode}
if sel := m.list.SelectedItem(); sel != nil {
    if item, ok := sel.(IssueItem); ok {
        // Suffix may contain dots (e.g. bt-ugbp.1); split before first hyphen only.
        ctx.CursorPrefix, _, _ = strings.Cut(item.Issue.ID, "-")
    }
}
if m.workspaceMode && m.activeRepos != nil {
    ctx.ActiveProjects = sortedRepoKeys(m.activeRepos)
}
return ctx
```

Empty `CursorPrefix` is the natural result for header rows, separators, empty lists, and unfocused-list states. `resolveHistoryPath` falls through to the filter branch, then the cwd fallback. No new helper function needed; the inline cut is one line.

## Coordination notes

### vs bt-08sh

bt-08sh migrates the correlator from JSONL+git-diff to Dolt-native. Two intersection points:

1. **Correlator signature**: bt-08sh likely changes what `NewCorrelator` takes. If it stops needing `repoPath`, Phase 3's `resolveHistoryPath` becomes vestigial for the bead-history case. Registry remains useful for adjacent features (bt-7l34, jump-to-files).
2. **Empty-state semantics**: bt-08sh's Dolt path doesn't have an "outside git repo" failure mode for bead history per se, so the registry-miss case shifts from "no history at all" to "no file-level git diffs." bt-08sh should preserve the empty-state branch for that.

No blocking dependency. Build u8iz now; bt-08sh adapts later.

### vs bt-m10p (paired beads)

Paired beads (e.g. `bt-zsy8` <-> `bd-zsy8`, same suffix across prefixes) work naturally on this primitive: each project (`bt`, `bd`) stamps its own prefix when bt is launched inside it. bt-m10p just looks up the paired prefix via `LookupAndValidate` and passes its registered path to a parallel correlator call. No changes required to the registry API when bt-m10p picks up.

The pair-detection-v1 false-positive concern (per project memory) is bt-m10p's design problem, not this primitive's — the registry holds whatever prefixes have been stamped; it doesn't infer pairs.

### vs bt-7l34 (firehose)

bt-7l34 needs to iterate the entire registry. The Registry type is already a `map[string]Entry`, so iteration is a one-line `for prefix, entry := range r {...}` on the consumer side. Skip a dedicated helper in Phase 1; bt-7l34 adds it if a wrapper proves useful.
