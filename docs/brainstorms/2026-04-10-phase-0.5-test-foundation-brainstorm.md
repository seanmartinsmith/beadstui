# Phase 0.5: Test Foundation Before Refactor

## What We're Building

A targeted pre-refactor pass that gets the test suite to a green baseline and documents what Phase 1 will break. This is not a test rewrite - it's scaffolding that makes the refactor safe.

## Why This Is Needed

The refactor plan (docs/plans/2026-04-09-refactor-charm-v2-cobra-migration-plan.md) says "each phase ends with green tests" but doesn't define the baseline. Two packages are already failing (cmd/bt, tests/e2e) due to a Dolt auto-detection bug. Without fixing this, we can't distinguish refactor regressions from pre-existing failures.

## Current Test Suite State

- 269 test files, 27 packages, ~1,500+ tests
- **25/27 packages green**, 2 red (cmd/bt, tests/e2e) - same root cause
- pkg/ui: 52 test files, 1,445 tests, 22.5s - the package Phase 1 will restructure
- Slow packages: pkg/export (17.3s), pkg/loader (20.9s), pkg/hooks (11s)

### Blast Radius Analysis (from research agent)

Phase 1's Model struct decomposition affects far fewer tests than expected:

| Category | Files | Impact |
|----------|-------|--------|
| Direct field access (`m.isBoardView`, `m.currentFilter`) | 5 files, ~36 occurrences | Must update |
| Direct `Model{}` struct literals | 1 file (sprint_view_keys_test.go), 17 occurrences | Must update |
| `NewModel()` constructor calls | 10 files, 91 calls | Safe if constructor signature preserved |
| Public accessor method tests | ~40 files | Zero impact |
| E2E tests (call binary, not struct) | 46 files | Zero impact |

**Bottom line: ~6 test files need explicit updates. ~45 work unchanged.**

### Test Architecture Layers

The test suite has proper layering - each layer tests the right thing:

- **Unit tests (pkg/ui/)**: In-memory `model.Issue` structs -> test UI logic, not data loading
- **E2E tests (tests/e2e/)**: JSONL fixtures in temp dirs -> test binary CLI/TUI behavior
- **Datasource tests (internal/datasource/)**: Test each backend (Dolt, SQLite, JSONL) properly
- **Component tests**: Individual sub-models (board, graph, tree, history) via public APIs

### Root Cause of Failing Tests

E2e tests create temp dirs with `.beads/beads.jsonl` fixtures. But bt's datasource discovery also probes `~/.beads/shared-server/dolt-server.port`. When a shared Dolt server is running (normal dev environment), it has priority 110 vs JSONL's priority 50. The shared server's global mode loads 2,308 issues across 14 databases, overriding the test fixtures.

This is an isolation bug, not a design flaw. JSONL fixtures are the right approach for e2e tests - they test TUI/CLI behavior, not the data layer.

## Key Decisions

1. **Scope: Targeted fix, not full audit.** Fix isolation, document baseline, add missing accessors. Full test audit happens after Phase 1 when the new structure is stable.

2. **Test audit timing: After Phase 1.** Phase 1 restructures pkg/ui (the biggest test package). Auditing before means auditing code about to change. After Phase 1, the new package structure is stable and we can optimize it properly.

3. **Dolt e2e tests: Separate workstream.** Important but doesn't block the refactor. Phases 1-3 don't touch the data layer. Tracked as a separate bead.

4. **Isolation fix approach: Respect BT_TEST_MODE.** E2e tests already set `BT_TEST_MODE=1`. Make datasource discovery skip shared Dolt server detection when this env var is set. Consistent with existing patterns (`BT_NO_BROWSER=1`).

5. **NewModel() is the test factory.** No need for a separate `testModel()` helper - `NewModel(issues, nil, "", nil)` is already used 91 times. Keep its signature stable through the refactor. Add accessor methods for fields that move into sub-structs.

6. **Fix the 6 files inline during Phase 1.** The 6 affected files are small enough to fix as each sub-struct is extracted. No separate pre-work needed.

## Phase 0.5 Scope (3 items)

### A. Fix e2e Dolt isolation (bt-hpq6)
- Make datasource discovery skip shared server probe when `BT_TEST_MODE=1`
- Gets cmd/bt and tests/e2e from red to green
- Establishes 27/27 baseline

### B. Document test baseline
- Snapshot: which packages pass, timing, test counts
- List the 6 files that Phase 1 will affect

### C. Verify accessor methods exist
- Check that public methods exist for all Model fields Phase 1 will move
- Add any missing ones (e.g., `CurrentFilter()`, `Issues()`)
- These are the stable API that keeps 40+ test files working through the refactor
